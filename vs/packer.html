Packer vs Boxfuse

Boxfuse is a simple, fast and robust alternative to Packer for creating AWS AMIs for JVM and Node.js apps.
Here's why you should consider switching...

Thinking of making the switch to Boxfuse to produce AWS AMIs for your JVM and Node.js apps?

We want to make the decision as easy as possible, so we’ve put together some info that'll help you decide which option is best for you.

Below, you'll find information to help you compare Boxfuse and Packer, including features, pricing and thoughts from current Boxfuse customers who have made the switch.

1. Boxfuse is easier and more robust than Packer

Boxfuse is all about making you productive so you can focus on delighting your customers instead dealing with infrastructure issues. We made sure Boxfuse is a joy to use by making building AWS AMIs for JVM application absolutely trivial. There are no lengthy complicated descriptors and no tedious OS provisioning. Instead all you need is a single command, boxfuse convert, which will take you all the way from JVM application to fully-baked AMI with no configuration required:

> boxfuse convert myapp-1.0.jar

Fusing Image for myapp-1.0.jar ...
Image fused in 00:06.525s (55362 K) -> myuser/myapp:1.0
Pushing myuser/myapp:1.0 ...
Verifying myuser/myapp:1.0 ...
Waiting for AWS to create an AMI for myuser/myapp:1.0 in eu-central-1 (this may take up to 50 seconds) ...
AMI created in 00:19.777s -> ami-e66264fb

Boxfuse does this by first analysing your application and then using that information to automatically generate an image with a minimal Linux-based OS on-the-fly. Boxfuse then uploads your new image to AWS and converts it to an AMI in the region of your choice.

Boxfuse runs on Windows, Mac and Linux and comes with special optimizations for popular tools and frameworks like Spring Boot, Dropwizard, Grails, Play, Executable Jars, Tomcat and TomEE.

2. Boxfuse is faster and images are smaller

At Boxfuse we strongly believe that being able to iterate quickly is a must. And that's why we've built the industry's fastest AMI creation process. You can literally go from a jar or war file on your local machine to a fully baked AMI in well under 60 seconds and sometimes even under 30 if AWS isn't under too much load.

Part of the reason why we manage to do that is size of our images. The minimal OS we generate is less than 1% of the size of a general purpose OS like Ubuntu. This results in much smaller AMI sizes which directly translates to faster instance boot times.

In fact this graph doesn't even do it justice as Boxfuse images are usually measured in the tens of MBs, but AWS shows them as 1 GB as this is their smallest supported EBS volume size.

We replaced Packer with Boxfuse which not only streamlined our deploy process, but also made it 2x faster. The Boxfuse Maven plugin made the integration extremely easy.

Tim Bart, CTO, Hello Inc.

3. Boxfuse is built for Continuous Deployment

Continuous Deployment is critically important for many companies today, allowing them to innovate and iterate faster.

Boxfuse fully embraces this by treating build tool plugins as first-class citizens, making it effortless to integrate AMI building into your pipeline. In addition to its command-line client, Boxfuse comes with both Maven and Gradle plugins which can be readily integrated through the usual means:

Gradle: build.gradle

plugins {
    id "com.boxfuse.client" version "{{site.boxfuseVersion}}"
}

Maven: pom.xml

<plugin>
    <groupId>com.boxfuse.client</groupId>
    <artifactId>boxfuse-maven-plugin</artifactId>
    <version>{{site.boxfuseVersion}}</version>
</plugin>

For rapid turnarounds and fast feedback, Boxfuse also lets you run your images on VirtualBox. This is a great way to run your integration tests locally before creating an AMI. And unlike Packer, Boxfuse's image are only created once and can be run unchanged on both VirtualBox and AWS, giving you perfect environment parity.
